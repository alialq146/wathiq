# 🛠️ رحلة بناء وثّق — سجل التطوير

توثيق تسلسلي لرحلة بناء منصة **وثّق** من تصميم ثابت إلى منتج حيّ كامل
(واجهة + قاعدة بيانات + ذكاء اصطناعي حقيقي).

**التاريخ:** ٣٠ يونيو ٢٠٢٦ · **المطوّر:** alialq146 · **الرابط:** https://wathiq-ai.vercel.app

---

## نظرة عامة

بدأنا من **حزمة تصميم** (نماذج HTML/CSS مُصدّرة من أداة تصميم) وانتهينا بتطبيق
**Next.js + TypeScript** منشور على Vercel، مربوط بقاعدة بيانات **Neon Postgres**،
ويستخدم **Claude (Opus 4.8)** لتحليل وثائق المتطلبات فعليًا.

---

## المراحل بالتسلسل

### المرحلة ١ — من التصميم إلى تطبيق حقيقي
- قراءة حزمة التصميم كاملة: `index.html` ومستورداته، التوكنز، و١٣ مكوّنًا.
- تحويل النماذج إلى تطبيق **Next.js (App Router) + TypeScript**.
- نقل **توكنز التصميم** (ألوان، خطوط، مسافات، تأثيرات) إلى CSS — عربي أولاً (RTL).
- نقل **١٣ مكوّنًا** إلى مكوّنات React مُوثّقة الأنواع.
- بناء **الهيكل + الشاشات الأربع** (نظرة عامة، المتطلبات، التفاصيل، تحليل وثّق).
- **التحقق:** `npm run build` نجح بدون أخطاء.

### المرحلة ٢ — الرفع على GitHub
- إنشاء مستودع `alialq146/wathiq`.
- حزم المشروع ورفعه (بدون `node_modules`).
- كتابة `README.md` ثنائي اللغة (عربي + إنجليزي).

### المرحلة ٣ — تأمين Next.js
- **المشكلة:** Vercel حظر النشر لأن إصدار Next.js فيه ثغرة أمنية.
- **الحل:** الترقية إلى `15.5.19` (أحدث إصدار مُصحَّح على نفس الجيل) → **٠ ثغرات**.

### المرحلة ٤ — النشر على Vercel
- استيراد المستودع في Vercel.
- 🔑 **نقطة مهمة:** التطبيق داخل مجلد فرعي `wathiq-app/` → ضبط **Root Directory = `wathiq-app`**.
- النشر.

### المرحلة ٥ — حل مشكلة الحظر (Deployment Blocked)
- **المشكلة:** Vercel (خطة Hobby) حظر النشر لأن **صاحب الـ commits** كان باسم/إيميل

- **الحل:** إعادة كتابة هوية كل الـ commits لتصبح باسمك `alialq146`، ثم دفع إجباري.

### المرحلة ٦ — قاعدة البيانات (Neon Postgres)
- إنشاء قاعدة **Neon** من تبويب Storage في Vercel وربطها بالمشروع.
- بناء طبقة بيانات بـ **Prisma** مع **تراجع آمن**: لو ما فيه قاعدة، التطبيق يشتغل
  ببيانات افتراضية بدل ما ينكسر.
- إضافة نقطة فحص `/api/health` تكشف مصدر البيانات (`database` أو `fallback`).
- سكربت بذرة **«مرة واحدة»** حتى لا تُمسح التعديلات عند إعادة النشر.

### المرحلة ٧ — نطاق مخصص
- إضافة النطاق النظيف **`wathiq-ai.vercel.app`** من إعدادات Domains.

### المرحلة ٨ — لوحة تعديل المتطلبات
- إضافة/تعديل/حذف المتطلبات من الواجهة عبر **Server Actions** تكتب في القاعدة.
- نافذة (Dialog) للنموذج + أزرار تعديل/حذف على كل بطاقة.

### المرحلة ٩ — تحليل وثّق بالذكاء الاصطناعي
- ربط التحليل بـ **Claude (claude-opus-4-8)** عبر `@anthropic-ai/sdk`.
- **مخرجات منظّمة (Structured Outputs)** لاستخراج المتطلبات بدقة.
- لصق وثيقة → استخراج المتطلبات + ملخّص شفّاف + خطوات الاستنتاج + توصيات + درجة ثقة.
- زر **«حفظ في القاعدة»** يضيف المستخرجة إلى Neon.
- المفتاح يُقرأ من متغيّر البيئة `ANTHROPIC_API_KEY` (لا يدخل الكود أبدًا).

### المرحلة ١٠ — رفع PDF للتحليل
- دعم رفع ملف **PDF** وتحليله مباشرةً عبر Claude (كتل محتوى مستند base64).
- واجهة سحب/إفلات مع حدّ حجم آمن يتوافق مع حدود Vercel للطلبات.

### المرحلة ١١ — البحث والتصفية والترتيب
- بحث برقم المتطلب أو نصّه، وفلترة بالحالة والأولوية، وترتيب — مع زر «مسح كل الفلاتر».
- ربط حقل البحث في الشريط العلوي بقائمة المتطلبات.

### المرحلة ١٢ — سير الاعتماد
- تفعيل زرّي **«طلب الاعتماد»** و**«طلب معلومات إضافية»** ليغيّرا حالة المتطلب فعليًا في القاعدة.
- حالة تحميل + رسائل نجاح/خطأ + تحديث فوري لشارة الحالة في التفاصيل والقائمة.

### المرحلة ١٣ — دورة المتطلب الكاملة والسياق وسجل التدقيق
- ربط معايير القبول/القواعد/الأسئلة بمتطلب محدّد وعرضها مفلترة.
- **إضافة معيار قبول**، **تبديل الإنجاز**، و**الإجابة عن الأسئلة** من الواجهة (Server Actions).
- **رؤى الذكاء الاصطناعي** صارت مبنية على بيانات المتطلب الحقيقية.
- بناء صفحات **أصحاب المصلحة** و**قواعد العمل** و**سجل التدقيق** بشكل حقيقي.
- **سجل تدقيق (AuditEvent)** يوثّق كل تغيير كخط زمني.
- تفعيل **مبدّل المشروع** و**قائمة الإعدادات** المنبثقة.

### المرحلة ١٤ — تصدير المتطلبات
- زر **«تصدير»** بثلاث صيغ: **PDF** (عبر طباعة المتصفح)، **Word** (‏.doc)، **Excel** (‏.csv).
- بدون مكتبات خارجية لضمان دعم العربية/RTL المثالي وخفّة الحزمة.
- تم التحقق البصري من مخرج PDF عبر لقطة شاشة (Chromium) قبل الدمج.

### المرحلة ١٥ — تسجيل الدخول
- صفحة دخول (بريد + كلمة مرور) بجلسة عبر **Cookie موقّع (HMAC)** + حماية بالـ **middleware**.
- **آمن بالتصميم:** المصادقة معطّلة ما لم تُضبط متغيّرات البيئة، فلا تُقفل الموقع بالخطأ.
- تحقّق شامل: تحويل غير المُصادَق إلى `/login`، رفض/قبول بيانات الدخول، ١١ اختبار وحدة، ولقطة شاشة للشاشة.

### المرحلة ١٦ — نظام الحسابات الكامل (v1.0.0)
- جدول **User** + صفحة **إنشاء حساب** `/signup` — أي شخص يسجّل ويعمل في مساحته.
- كلمات المرور مشفّرة بـ **scrypt**، والجلسات بـ Cookie موقّع يحمل هوية المستخدم.
- **عزل البيانات:** `ownerId` على كل الجداول؛ كل حساب يرى بياناته + التجريبية المشتركة، وحُماية الملكية مفروضة في كل Server Action.
- سجل التدقيق يوثّق العمليات **باسم منفّذها الحقيقي**، والشريط الجانبي يعرض الحساب المسجّل.
- **اختبار غير مسبوق:** شغّلنا Postgres حقيقيًا داخل بيئة التطوير، ثم متصفح آلي أنشأ حسابًا ومتطلبًا من الواجهة نفسها، وتحققنا في القاعدة من الملكية والعزل والتشفير — قبل أي اعتماد.

### المرحلة ١٨ — جاهزية النشر (v1.2.0)
- **خطة مجانية** (تحليل واحد) + بطاقة ترقية عبر **واتساب** + صفحة أسعار `/pricing`.
- **وضع داكن** كامل عبر توكنات المظهر + زر تبديل بلا وميض.
- **صفحات قانونية** (خصوصية/شروط) + **Favicon** و**OpenGraph**.
- **المستخدم الجديد يبدأ فارغًا** + شاشة نظرة عامة مبنية على بيانات حقيقية.
- **توافق كامل مع الجوال** (قائمة منزلقة + انسدال لوحة التفاصيل + شبكات متكيّفة).
- تحقّق بصري شامل (سطح مكتب/جوال، فاتح/داكن) عبر لقطات Chromium.

### المرحلة ١٩ — الباقات والمشاريع وتحليل المتطلب (v1.3.0)
- **٣ باقات** (مجاني/احترافي/الأعمال) + حقول اشتراك + بطاقة استخدام في اللوحة + بطاقات ترقية.
- **تحليل على مستوى كل متطلب**: درجة جودة، حالة، غموض/مخاطر، أسئلة، معايير قبول، SMART، وصياغة محسّنة — عبر توسيع خدمة التحليل نفسها (`/api/analyze-requirement`). نافذة الإضافة صارت للمدخلات فقط.
- **المشاريع**: إنشاء مشروع فعلي + مبدّل + عزل كامل بين المشاريع + قيود حسب الخطة + ترحيل آمن للمستخدمين الحاليين.
- تحقّق شامل: أنواع + بناء + وحدة + E2E على Postgres حقيقي (مجاني/احترافي، عزل، حدود، ملكية).

### المرحلة ٢٠ — توجيه النماذج وتتبّع الاستخدام وتقييم SMART (v1.4.0)
- **توجيه النماذج حسب الباقة**: مجاني → Haiku، احترافي → Sonnet، الأعمال → Opus — قابل للضبط عبر `AI_MODEL_*` بلا تعديل كود. الاختيار **في الخادم فقط** حسب خطة المستخدم الحقيقية.
- **تتبّع الاستهلاك**: جدول `AiUsage` يسجّل النموذج والرموز والتكلفة التقديرية والحالة (`SUCCESS`/`FAILED`/`BLOCKED_LIMIT`/`BLOCKED_SIZE`)؛ تسجيل best-effort والفاشل لا يُحتسب من الرصيد.
- **فحوصات مسبقة** قبل كل استدعاء لـ Claude: مصادقة، حدّ، حجم، وملكية المشروع/المتطلب.
- **إعادة تعيين شهرية** لرصيد التحليلات عند أول استدعاء بعد `resetDate`.
- **تحسينات التحليل**: SMART ثلاثي الحالة (مستوفٍ/جزئي/غير مستوفٍ) مع سبب، أنواع متطلب واضحة، وحقل ملاحظات اختياري يُمرَّر للذكاء الاصطناعي.
- تحقّق: أنواع + بناء + وحدة (توجيه ٨/٨، إعادة تعيين ٦/٦) + تسجيل استخدام على Postgres حقيقي.

### المرحلة ٢١ — لوحة الأدمن v1 (v1.5.0)
- **`/admin` مستقلة** بسبعة تبويبات (نظرة عامة، مستخدمون، اشتراكات، AI Usage، أخطاء، إعدادات، صحة النظام) — RTL + وضع داكن، بلا مكتبات، وchunk منفصل لا يمس تجربة المستخدم.
- **أدوار**: `role` و`accountStatus` بقيم افتراضية آمنة؛ التعيين يدويًّا فقط عبر `scripts/make-admin.mjs`؛ الدور يُقرأ من القاعدة في كل طلب (لا من الكوكي).
- **إدارة مستخدمين**: خطة/عداد/حد مخصص (`limitOverride` يحترمه فحص الحد الفعلي)/تعطيل-تفعيل/تفاصيل — كلها عبر API محمي بـ SUPER_ADMIN.
- **مراقبة AI**: توزيع حسب النموذج، أعلى المستهلكين، سجل بفلاتر، وأخطاء مع **اقتراحات تشخيصية** («ماذا أفحص؟»).
- **مالية تقديرية**: MRR/ARR، تكلفة AI (USD→SAR عبر `USD_TO_SAR`)، هامش لكل خطة.
- **أمان**: الأسرار Configured/Missing فقط؛ لا محتوى ملفات في السجلات؛ منع تعطيل الذات.
- تحقّق: أنواع + بناء + ٣٨ فحص E2E على Postgres حقيقي + لقطات لكل التبويبات.

### المرحلة ٢٢ — Requirement Workspace v2 + جولة الجودة (v1.7.0)
- **حقول دورة الحياة**: مصدر المتطلب، المسؤول، ورقم الإصدار — والتفاصيل صارت ملف عمل متكاملًا.
- **مساعد وثّق**: الذكاء الاصطناعي صار مساعدًا اختياريًا معلَنًا (٤ خيارات) بدل خطوة تبدو إجبارية؛ والمصطلح توحّد على «مؤشر الجودة».
- **ملخص مشروع بلا AI**: توزيع الأولويات والحالات محسوب من الحقول مباشرة.
- **تعليقات أمنية عربية** في نقاط الحراسة (Server Actions، middleware، توجيه النماذج، الفحوصات المسبقة) ليسهل على أي مراجع فهم «لماذا».
- ملحوظات من مراحل سابقة ضمن نفس الجولة: v1.5.1–1.5.3 (نصوص الهبوط والأسعار والمصادقة) وv1.6.0 (Onboarding وEmpty States وإزالة النصوص التجريبية المضللة).
- تحقّق: أنواع + بناء بعد كل مرحلة + E2E على Postgres حقيقي بلقطات.

### المرحلة ٢٣ — مخرجات احترافية ومساعد أذكى (v1.8.0)
- **تقرير تحليل متطلبات** رسمي RTL (غلاف/ملخص تنفيذي/جدول/تفاصيل/توصيات/خاتمة) بثلاث صيغ PDF/Word/CSV، مع نافذة خيارات (نوع، صيغة، محتويات، **نطاق النتائج المفلترة**) — بلا مكتبات، وأُصلح غلافٌ كان يعرض بيانات المشروع التجريبي.
- **سجل تغييرات المتطلب** Timeline بفروقات حقلية («الأولوية من متوسطة إلى عالية؛ تعديل الوصف؛ V1 ← V2») ورفع إصدار تلقائي عند تغيّر المحتوى — عبر سجل التدقيق الموجود بلا جدول جديد.
- **فلاتر متقدمة** (نوع/مصدر/مسؤول/محلَّل/معايير قبول) بقيم مشتقة من البيانات + chips نشطة + ربط بالتصدير.
- **مساعد وثّق v2**: ٦ مهام؛ الخفيفة منها بـ prompts مركّزة وschemas مصغّرة (تكلفة أقل بكثير)، ونتائجها بطاقات قرار (اعتماد/إضافة/حفظ كملاحظة/تجاهل) بنفس فحوصات الحد والملكية والتسجيل.
- تحقّق كامل على Postgres حقيقي بلقطات (تقرير مفلتر، سجل، فلاتر، جوال) — ومنع المساعد عن متطلبات الغير.

### المرحلة ٢٧ — تدقيق الجودة الشامل قبل الإطلاق (v1.9.5)
- QA Release Audit كامل بلا ميزات جديدة: مراجعة كود ثلاثية المحاور + أكثر من ٩٠ فحص E2E على الباقات الثلاث والمساعد والتصدير والفلاتر والسجل والأدمن والعزل والجوال — كل المحاور الأساسية خضراء.
- حيلة اختبار مفيدة: مفتاح AI وهمي + منفذ ميت + أسماء نماذج اختبارية في `AI_MODEL_*` أثبتت توجيه النماذج والحصص وتسجيل الفشل **بدون أي تكلفة API**.
- أُغلقت ثغرتان متوسطتان (جلسة مستخدم محذوف/معطل تتجاوز القياس والملكية؛ اسم نموذج داخلي ظاهر لكل المستخدمين) وثلاث ملاحظات منخفضة — والباقي وُثّق كتوصيات مصنفة في تقرير الإطلاق.

### المرحلة ٢٦ — البقاء داخل صفحة المتطلب + تبويب «تقييم الجودة» (v1.9.4)
- إصلاح تجربة ما بعد المساعد: كان المستخدم يُعاد للقائمة الرئيسية بعد نجاح التحليل لأن تحديث البيانات يعيد بناء الواجهة — الآن تُحفظ الشاشة والمتطلب في `sessionStorage` ويُشتق المتطلب من أحدث بيانات، فتبقى المهام الست كلها داخل نفس الصفحة.
- **تبويب «تقييم الجودة»** جديد يعرض النتائج التفصيلية (SMART، الغموض والمخاطر، الصياغة المحسّنة مع الاعتماد)، واللوحة الجانبية صارت ملخصًا سريعًا وأزرار مهام فقط مع زر «عرض النتائج التفصيلية».
- رسالة نجاح واضحة «تم تحديث نتائج مساعد وثّق بنجاح.» مع فتح التبويب تلقائيًا، ورسالة خطأ بلا إخراج «تعذر تشغيل مساعد وثّق. حاول مرة أخرى.» — وحفظ التبويب النشط لكل متطلب حتى بعد إعادة التحميل.
- تحقّق: ٢١ فحص E2E بمحاكاة استجابات المساعد (نجاح/مهام خفيفة/خطأ/إعادة تحميل/حالة فارغة) + بناء نظيف.

### المرحلة ٢٥ — سلسلة إصلاحات تقييم الجودة في الإنتاج (v1.9.1 – v1.9.3)
- ثلاث طبقات أعطال متتالية ظهرت في الإنتاج وأُصلحت واحدة تلو الأخرى: **انقطاع JSON عند حد الرموز** (رفع الحدود + مستخرج موحّد بأخطاء مصنّفة في لوحة الأدمن) ← **قتل الدالة عند مهلة ٦٠ ثانية** (رفعها إلى ٣٠٠ + حدود إيجاز في التعليمات) ← **انتهاء معاملة Prisma على Neon** (دفعات `createMany` + مهلة معاملة صريحة).
- درس مستفاد: كل إصلاح كشف الطبقة التالية — التشخيص من `AiUsage.errorMessage` ولوحة الأدمن اختصر الطريق.

### المرحلة ٢٤ — وثائق BRD و SRS (v1.9.0)
- نافذة التصدير صارت تختار **نوع المستند** (تقرير تحليل / BRD / SRS) بوصف تحت كل نوع، مع PDF وWord قابل للتعديل والنطاق المفلتر.
- **BRD** بـ ١٥ قسمًا و**SRS** بـ ١٥ قسمًا (مع RTM وحالات استخدام مستنتجة موسومة «مقترحة») — كلها من بيانات المشروع الحقيقية فقط، والنواقص تُوسم صراحة «يحتاج استكمال» بدل الاختراع.
- `report-config.ts` يجمع كل النصوص القابلة للتغيير، وبنية «تحسين الصياغة بالمساعد» جاهزة لـ v2.0 (معطَّلة) — الوثائق لا تستهلك أي ذكاء اصطناعي.
- تحقّق: توليد فعلي من مشروع حقيقي بلقطات + فحص آلي (لا نماذج مذكورة، أوسمة النقص تعمل) + جوال + بناء نظيف.

### المرحلة ١٧ — صفحة الهبوط (v1.1.0)
- بناء **صفحة هبوط عربية احترافية** بمستوى SaaS عالمي (Hero، مميزات، كيف يعمل، لقطات حقيقية بتكبير، أرقام متحركة، أسئلة شائعة، CTA، تذييل) في مكوّن واحد بدون مكتبات.
- التقطنا **لقطات حقيقية** من النظام (عبر متصفح آلي) لاستخدامها في الصفحة.
- `/` صار يوجّه: الزائر → صفحة الهبوط، والمسجّل → مساحة العمل.
- تحصين أمني: حارس في كل Server Action يرفض الطلبات غير المصادَقة بعد فتح `/` للعموم.
- تحقّق بصري (سطح المكتب/جوال) واختبار توجيه كامل.

## 📚 دروس مستفادة

1. **متغيّرات البيئة تنطبق على النشرات الجديدة فقط** — بعد إضافة أي متغيّر، لازم
   **Redeploy** (ويُفضّل بدون Build Cache).
2. **الأسرار لا تدخل الكود** — المفاتيح (قاعدة البيانات، Claude) تُحفظ كمتغيّرات بيئة،
   خصوصًا والمستودع عام.
3. **Vercel Hobby = استخدام فردي** — صاحب المستودع + صاحب الـ commits + صاحب حساب
   Vercel لازم يكونون نفس الشخص، أو يكون المستودع عامًّا.
4. **التراجع الآمن يمنع الأعطال** — صمّمنا التطبيق ليعمل حتى لو غابت القاعدة أو المفتاح.
5. **التصميم الجيد للمجلدات** — التطبيق في `wathiq-app/`، فيلزم ضبط Root Directory.
